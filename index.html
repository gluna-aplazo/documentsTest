Estrategia de Automatización 2026

Tres hitos para automatizar la cobertura y regresión de todos los flujos

Cada hito representa un paso concreto hacia el objetivo principal: eliminar la intervención manual del QA Engineer en la ejecución de pruebas, haciendo que la calidad corra de forma autónoma y continua.

🎯

Objetivo principal

Automatizar la ejecución de todos los flujos de cobertura y regresión, eliminando la intervención manual del QA Engineer, garantizando validaciones continuas, trazabilidad completa y visibilidad inmediata de resultados en cada cambio del sistema.

Contexto

¿Por qué necesitamos estos hitos?

Actualmente la ejecución de pruebas de cobertura y regresión depende de disparos manuales por parte del QA Engineer. Esto genera brechas de validación, falta de visibilidad y retrasos en la detección de defectos. Los tres hitos que se describen a continuación atacan este problema en capas: el primero integra las pruebas al pipeline de liberación, el segundo las conecta al flujo natural del ticket en Jira, y el tercero cierra el ciclo validando el estado real de los datos en base de datos.

Detección temprana

Defectos detectados antes de llegar a producción, sin que el QA tenga que intervenir.

Trazabilidad completa

Cada cambio queda relacionado con su cobertura ejecutada y el resultado obtenido.

Visibilidad inmediata

Los equipos reciben notificaciones automáticas en Slack con el resultado de cada ejecución.

Menos carga manual

El QA Engineer se enfoca en diseño y análisis, no en ejecutar y monitorear pruebas.

Hitos hacia el objetivo

Los tres hitos

Cada hito es independiente pero complementario. Juntos cubren el ciclo completo: liberación → ticket → datos.

1
🚀 Hito 1 · Pipeline CI/CD Propuesto

Automatización desde pipelines de liberación

Incorporar un stage dentro del Jenkinsfile de cada proyecto. Al ejecutarse un pipeline en Stage o al realizarse una liberación, ese stage dispara automáticamente las pruebas asociadas al componente según la matriz de trazabilidad, sin ninguna acción del QA Engineer.

¿Qué resuelve este hito?

Garantiza que ninguna liberación llegue a Stage sin haber ejecutado su cobertura correspondiente. El QA no necesita estar presente para que las pruebas corran.

Funcionamiento

El developer hace push o ejecuta el pipeline en Stage.
El stage en el Jenkinsfile identifica el servicio y consulta la matriz de trazabilidad.
Se invocan automáticamente las suites vinculadas al componente.
Al concluir, se envía notificación a Slack con resultado, build number y enlace a evidencia.

2
✅ Hito 2 · Jira Automation ✅ Implementado

Automatización por cambio de estado en Jira

Reglas en Jira Automation que disparan las pruebas de cobertura automáticamente cuando un ticket transita de In Progress a Ready For QA. El QA Engineer recibe el resultado en Slack sin haber ejecutado nada manualmente.

¿Qué resuelve este hito?

Integra la validación al flujo natural de trabajo del equipo. Cada vez que un desarrollador termina una tarea, las pruebas correspondientes se ejecutan solas.

Disparador de estado

In Progress
→
Ready For QA ⚡
Dispara automatización

Funcionamiento

El ticket transita de In Progress a Ready For QA en Jira.
La regla identifica el servicio o componente afectado.
Según la matriz de trazabilidad, se disparan las pruebas del servicio correspondiente vía Lambda.
El resultado llega automáticamente al hilo de Slack: ✅ SUCCESS o ❌ FAILURE con build number y duración.

3
🔮 Hito 3 · API de datos Propuesto

API de validación de datos alineada al North Star

Un nuevo servicio que realiza consultas directas a la base de datos PostgreSQL según los parámetros del caso de prueba. Esto permite a las automatizaciones validar el estado real de los datos en cada ejecución, sin intervención del QA para verificar manualmente la BD.

¿Qué resuelve este hito?

Cierra el ciclo completo de calidad: ya no basta con validar la UI o la respuesta del API; este hito confirma que el dato quedó correctamente persistido en base de datos, eliminando la verificación manual post-ejecución.

Alineación con el North Star de Aplazo

Este servicio habilita validaciones de datos reales en cada prueba automatizada, cerrando el ciclo entre la ejecución de pruebas UI/API y la verificación del estado final en base de datos.

Validación de datos reales Trazabilidad end-to-end PostgreSQL directo Parametrizable por request

Funcionamiento

La automatización envía un request con los parámetros del caso de prueba (usuario, transacción, estado esperado).
El controller valida los parámetros y ejecuta la query correspondiente en PostgreSQL.
El resultado se devuelve al test para validar que el estado en BD sea el esperado.
Si hay discrepancia, el test falla con contexto claro del dato incorrecto, sin necesidad de inspección manual.

Cómo se construyó el Hito 2

Historial de implementación

Jira Automation Rule
Regla con trigger "Issue status changed" de In Progress a Ready For QA y acción Send web request apuntando a la Lambda.
Slack App — Webhook Notificaciones
App con scopes chat:write, groups:history y channels:read para postear y leer el canal privado.
AWS Lambda — jira-slack-bot-trigger
Lambda en Node.js 20.x con Function URL pública. Recibe el payload de Jira, identifica el servicio, postea en Slack, dispara Jenkins y hace polling del resultado cada 10 segundos.
Matriz de trazabilidad en FLOW_MAP
La Lambda contiene el mapeo servicio → tag → job. Al recibir el ticket identifica qué suite ejecutar según el componente afectado.
Resultado en Slack
El mensaje se actualiza automáticamente en el hilo con ✅ SUCCESS o ❌ FAILURE, incluyendo build number, duración y link a Jenkins.

Requerimientos y dependencias

¿Qué necesitamos para avanzar con los hitos?

Para completar los tres hitos se requiere el apoyo coordinado de tres equipos clave. Sin estos apoyos, la implementación no puede avanzar de forma segura ni sostenible.

1
Apoyo de Developers
1 dev por squad
  • Cambios en los Jenkinsfiles de los repositorios involucrados.
  • Apoyo con la configuración y despliegue de Lambdas.
  • Pruebas unitarias que garanticen que los cambios no afecten el comportamiento actual de los repositorios.
2
QA Tooling Engineers
Mantenimiento y gestión
  • Mantenimiento y gestión de Lambdas de automatización.
  • Actualización de frameworks y librerías para garantizar la correcta ejecución de los casos de prueba existentes y futuros.
  • Soporte técnico continuo sobre las herramientas de automatización del ecosistema Aplazo.
3
Apoyo de DevOps
Accesos y configuración
  • Acceso o modificación a repositorios necesarios para implementar los cambios.
  • Gestión de Lambdas en el entorno de infraestructura de Aplazo.
  • Permisos y configuración en Jenkins para los nuevos jobs y stages.
  • Acceso a las tablas de BD en Stage para las validaciones del Hito 3.
💡 Nota: Estos apoyos son transversales a los tres hitos. Contar con ellos desde el inicio garantiza una implementación segura y sostenible, sin generar deuda técnica en los repositorios existentes.
Conclusión

Una estrategia de calidad continua y end-to-end

Los tres hitos son complementarios y cubren el ciclo completo de calidad. El Hito 1 valida en el momento de la liberación, el Hito 2 integra la cobertura al flujo natural del ticket en Jira —ya implementado y funcionando—, y el Hito 3 cierra el ciclo con validaciones directas sobre la base de datos. Juntos eliminan la dependencia del QA Engineer para ejecutar, monitorear o verificar pruebas, y convierten la calidad en un proceso continuo y autónomo dentro de Aplazo.
